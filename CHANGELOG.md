## 语义化版本（Semantic Versioning，简称 SemVer）
1. MAJOR（主版本号）：当你做了不兼容的 API 变更（如自动化测试框架核心接口重构、测试用例格式大幅调整、无法向下兼容旧版本配置）时，递增主版本号（如 1.3.2 → 2.0.0）。
2. MINOR（次版本号）：当你做了向后兼容的功能新增（如新增测试报告导出格式、支持新的被测系统协议、增加批量执行用例功能）时，递增次版本号，修订版本号重置为 0（如 1.3.2 → 1.4.0）。
3. PATCH（修订版本号）：当你做了向后兼容的问题修复（如修复测试用例执行失败的 BUG、优化镜像启动速度、修正报告数据错误）时，递增修订版本号（如 1.3.2 → 1.3.3）。
4. 可选扩展（按需使用）：
    - 预发布版本：MAJOR.MINOR.PATCH-[预发布标识]（如 1.4.0-alpha.1、2.0.0-beta.3，用于测试环境灰度验证）。
    - 构建信息：MAJOR.MINOR.PATCH+[构建编号/哈希]（如 1.3.2+build123、1.4.0-alpha.1+a8f7d2，用于关联具体代码提交）。
### 预发布版本标识

| 标签    | 含义               | 示例             | 适用场景         |
|-------|------------------|----------------|--------------|
| alpha | 早期内部测试版，功能可能不全   | v3.0.0-alpha.1 | 开发团队内部验证核心功能 |
| beta  | 公测版，功能基本稳定但可能有bug | v3.0.0-beta.2  | 邀请社区用户参与测试   |
| rc    | 候选发布版            | v3.0.0-rc.2    | 功能冻结，仅修复关键问题 |

规则：
- 预发布版本号按 `.0`、`.1`、`.2` 顺序递增
- 正式版本发布后，预发布标签移除（如 v3.0.0-rc.3 -> v3.0.0）
